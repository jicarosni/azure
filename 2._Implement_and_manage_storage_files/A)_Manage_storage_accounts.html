Implement Azure storage replication

Azure Storage redundancy

Microsoft recommends using ZRS in the primary region for Azure Data Lake Storage Gen2 workloads.

1. Locally redundant storage (LRS)

- copies your data synchronously three times within a single physical location (single data center) in the primary region.
- LRS is the least expensive replication option, but is not recommended for applications requiring high availability.
- (11 nines) durability of objects over a given year.
- write request to a storage account that is using LRS happens synchronously
- the write operation returns successfully only after the data is written to all three replicas.

2. Zone-redundant storage (ZRS)

- copies your data synchronously across three Azure availability zones in the primary region.
- for applications requiring high availability, Microsoft recommends using ZRS in the primary region, and also replicating to a secondary region.
- 12 9's
- write request to a storage account that is using ZRS happens synchronously
- scenarios that require consistency, durability, and high availability.
- may NOT protect your data against a regional disaster where multiple zones are permanently affected


3. Geo-redundant storage (GRS)

- copies your data synchronously three times within a single physical location in the primary region using LRS.
- then copies your data asynchronously to a single physical location in the secondary region
- within the secondary region, your data is copied synchronously three times using LRS.
- 16 9's

4. Geo-zone-redundant storage (GZRS)

- copies your data synchronously across three Azure availability zones in the primary region using ZRS
- then copies your data asynchronously to a single physical location in the secondary region
- within the secondary region, your data is copied synchronously three times using LRS.
- 16 9's

- Only general-purpose v2 storage accounts support GZRS and RA-GZRS
- GZRS and RA-GZRS support block blobs, page blobs (except for VHD disks), files, tables, and queues.
- When read access to the secondary is enabled, your application can be read from the secondary endpoint as well as from the primary endpoint.
- primary endpoint for Blob storage is myaccount.blob.core.windows.net
- secondary endpoint is myaccount-secondary.blob.core.windows.net
